# LEARNINGS.md — hard-won lessons from the Phase D build (2026-07-14/15)

Read this before making changes. Every item here cost real debugging time.

## Architecture traps (will bite again in the Shopify port)

1. **Never let a framework-free widget self-mount into a React-owned div.**
   The widget mounted on `window.load`; React hydration finishing later wiped it, so customers saw no reviews, but only in prod (locally hydration always won the race). Fix that stuck: host page marks the div `data-reviewos-manual` and calls `window.ReviewOS.mount()` from `useEffect`; the mount is idempotent keyed off DOM presence (`.rvos-widget` child), not a data-attribute flag (flags survive wipes and block remounts). Plain non-React embeds still auto-mount. The Shopify Theme App Extension embed is non-React, so auto-mount path must keep working.

2. **Inline `onclick="event.stopPropagation()"` breaks delegated event handling.**
   The whole widget uses one delegated click listener on the root. Someone (an agent) added stopPropagation inside the modal to stop backdrop-close from firing on inner clicks; that silently killed the star picker and close button, meaning NO review could ever be submitted, and it passed several "modal opens" verifications. Correct pattern: backdrop elements get a guard in the delegated handler (`if (isBackdrop && evt.target !== target) return`), never stopPropagation.

3. **`esc()` (HTML entity escaping) is NOT safe inside a JS-string-in-attribute context** (`onerror="...'${esc(x)}'..."`), because the browser decodes entities before the JS parses. Never interpolate data into inline handlers; pre-render a hidden fallback sibling and toggle it with static JS.

4. **`history.replaceState` for filter URL sync makes the Back button exit the page.** Users expect Back to undo a filter. Use `pushState` per user-driven change plus a `popstate` handler that re-reads the URL and refetches.

5. **Prisma cannot switch datasource provider per environment.** "SQLite in dev, Postgres in prod" is not a real option with one schema. We moved everything (local dev included) to Neon Postgres. Old sqlite migrations were deleted; schema is managed with `prisma db push` + seed. If migration history is needed later, re-baseline on Postgres.

## Shopify app / React Router traps (Phase 2, 2026-07-14)

21. **A `.server` module's runtime VALUE must never be imported into a route's client component.** React Router strips server code only from `loader`/`action`/`middleware`/`headers`. If the default-export component (or a child component in the same file) references a runtime value from a `*.server.ts` module, RR errors "Server-only module referenced by client", the route fails to load, and App Bridge bounces the merchant to the app home. `typeof x` type queries are fine (erased); only runtime values break it. Fix: put shared client+server constants/enums in a plain (non-`.server`) module with no Prisma import. Caught only by `shopify app dev`/`npm run build`, NOT by `tsc`/typecheck. Bit us on `REVIEW_STATUSES` in `app.reviews.tsx`; will recur in Phase 3 proxy/widget routes.
22. **This template uses Polaris WEB COMPONENTS (`s-page`, `s-badge`, `s-select`, `s-text-area`, ...), not React `@shopify/polaris`.** No `@shopify/polaris` dep exists. Unknown `s-*` attributes fail silently (don't throw); use the `shopify-polaris-app-home` skill's component list for valid tags/props.
23. **iCloud-synced `~/Desktop` creates `<name> 2/` duplicate dirs on sync conflicts.** One appeared inside `prisma/migrations/` (empty, untracked) and broke `prisma migrate deploy` with P3015 ("could not find migration.sql"). These won't show in `git status` (untracked/ignored) — check the raw filesystem when a tool complains about a file/dir you didn't create.

## Ops / deploy traps

6. **Piping a secret into `vercel env add` leaves a trailing newline in the value.** Every Groq fetch then threw `TypeError: Headers ...` and the API 500'd. Use `printf '%s'`, never `echo | `.
7. `@vercel/react-router` preset only supports React Router 7; this app is on 8. Zero-config Vercel detection works, use that. Re-check the preset when it supports RR8.
8. `vercel install neon` needs `--plan free_v3` (not `free`) and a one-time browser terms acceptance by the user.
9. Vite dev server aborts in-flight widget fetches on its first-load dependency-optimization reload, producing a one-off `[reviewos] init failed TypeError: Failed to fetch`. Dev-only noise; does not happen in prod. Don't chase it again.

## Process lessons (agent orchestration)

10. **"Verified" from a subagent means "the checks it thought of passed."** The write-review modal was reported working in D2, D4, and D5 verification passes because they only checked "modal opens." The bug was found only by a QA agent told to exercise every interaction end-to-end (click each star, submit, check the value). Verification prompts must enumerate interactions, not features.
11. **A bug that reproduces once in prod and never locally is still a real bug.** The mount wipe showed up in 1 of ~10 live loads and no local runs. The root cause was reasoned from evidence (trust-badge image requests fired, DOM empty after), not from a reliable repro. Timing-dependent races deserve an architectural fix, not more retries of the repro.
12. Session limits kill background agents mid-task; their partial edits stay in the working tree. Always `git status && git diff` after an agent dies before assuming anything about repo state.
13. Static-audit agent + live-QA agent in parallel find disjoint bug sets (audit found API validation holes QA couldn't see; QA found the dead modal the audit's code-reading missed because the stopPropagation looked intentional). Use both for any "find bugs" request.

## Data / demo state

14. The live demo shares one Neon DB with local dev. Submitted reviews persist for everyone; a "QA TEST - ignore" review sits in the moderation queue (status pending). Reseeding (`npm run seed`) wipes and rebuilds everything including AI summary cache (regenerated on demand, cached in AiSummary).
15. GROQ_API_KEY was exposed in chat earlier and is now in Vercel prod env. ROTATE it at console.groq.com, then `vercel env rm/add` (with printf!) and redeploy.

## Shopify scaffold traps (Phase 1, 2026-07-14)

16. **Theme APP-EXTENSION blocks cannot use `{% stylesheet %}` / `{% javascript %}` tags** (theme-check `AppBlockValidTags`) — those are for THEME blocks only. Use inline `<style>` / `<script>` in the block liquid, or the `stylesheet`/`javascript` schema keys pointing at `assets/`. Critically, `shopify app config validate` does NOT catch this; only `shopify app dev`'s theme-check does. Never trust config-validate as proof a block works — run `shopify app dev`.
17. **`shopify app init --path X --name Y` nests the app at `X/Y/` and gives it its own `.git`.** Flatten to the intended dir and `rm -rf` the nested `.git` so the parent repo tracks it (else it looks like an embedded submodule).
18. **Shopify CLI ink prompts (e.g. storefront password) CANNOT be driven by `expect`/PTY** — neither `\r` nor `\n` submits; keystrokes just accumulate in the field. Run `shopify app dev` once interactively in a REAL terminal to answer the password + do the browser install; the CLI caches both, so later runs work from a background shell with no prompt. `shopify app init`/`generate extension` DO work non-interactively with full flags (`--template reactRouter --flavor typescript`, `-t theme_app_extension`).
19. **App Proxy path mapping**: storefront `/apps/{subpath}/{rest}` → `{proxy_url}/{rest}`. Set `[app_proxy].url` in toml with a `/proxy` suffix and route under `app/routes/proxy.*` (flat-routes). `shopify app dev` auto-rewrites `proxy_url` to `<tunnel>/proxy` at runtime (local toml keeps the `example.com/proxy` placeholder — `shopify app deploy` will need the real host).
20. **Dev store storefront is password-protected**; the App Proxy curl 302-redirects to `/password`. Bypass without disabling: POST `form_type=storefront_password&password=<pw>` to `/password` with a cookie jar, then GET the proxy URL with that jar.
21. **Never place `.test.ts` files under `app/routes/`** — React Router's file-route convention loads them as route modules, and the top-level `vitest` import crashes the dev server with a 500 on EVERY request (widget showed "Couldn't load reviews" with zero code bugs). Typecheck and vitest both stay green, so only running the live server exposes it. Tests live in `app/lib/` or `test/`.
22. **Parallel implementation subagents drift on shared API contracts** — two agents built proxy routes and the widget client with mismatched response keys, param names, and id-vs-handle semantics; both sides typechecked and unit-tested green. Pin the exact contract (params + response shapes) verbatim in BOTH prompts, then run a seam-focused diff review plus an integration test using the consumer's exact shapes.
23. **Seed `shop` must match the real dev store domain** (`SHOP_DOMAIN=reviewos.myshopify.com npm run seed`), and storefront product linkage resolves by `Product.slug == Shopify product handle` — seed slugs won't match real products until catalog sync exists (Phase 4). Dev shim: remap one seed product's slug to a real handle.
24. **Block `{% schema %}` `name` max 25 chars** — theme-check rejects longer names only at `shopify app dev` time.
25. **Neon P1001 ("can't reach database server") on `shopify app dev` startup** is usually a scale-to-zero cold-start blip: retry once or add `connect_timeout=15` to DATABASE_URL.
26. **Importing `shopify.server` (which calls `shopifyApp({...})` at module load) into the proxy/services import graph breaks unit tests.** `shopifyApp` throws "Detected an empty appUrl configuration" at import time when app env isn't set, so any vitest that imports a proxy route transitively 500s at collection. Phase 4's `products.server.ts` needed `unauthenticated` from `shopify.server` for storefront auto-create; a static import broke 4 contract tests. Fix: **lazy `await import("../shopify.server")` inside the function** that needs it (only runs on a real cache-miss), keeping shopifyApp init out of the module-load graph. Bonus: proxy cold starts don't pay for shopifyApp init on cache hits. tsc stays green; only vitest catches it.
27. **`npx` is broken in this shell** — the `rtk hook claude` PreToolUse rewrite errors with `[rtk: No such file or directory]` on `npx ...`. Call binaries directly instead: `./node_modules/.bin/prisma`, `./node_modules/.bin/vitest run`, `./node_modules/.bin/tsc --noEmit`, `./node_modules/.bin/shopify app build`. (Fix the rtk hook config later.)
29. **`process.env.X ?? "default"` does NOT fall back on an empty string.** `.env` had `AI_PROVIDER=` (empty), so `getAiProvider()`'s `?? "groq"` selected provider `""` and threw `unknown AI_PROVIDER:` on *every* summary generation — silently, because the proxy loader's error became a widget catch → `aiSummary:null` → block hides itself. Symptom looked like "AI summary block broken" but was a dead AI provider. Use `|| "groq"` (or trim + check) whenever an env var can be present-but-empty. tsc/build/vitest all stayed green; only exercising generation against real env caught it.
30. **Real store products start with zero reviews; seed data lives on fictional slugs.** After product sync, a real Shopify handle (e.g. `the-collection-snowboard-liquid`) has a Product row but no reviews, so summary/feed are empty and the AI-summary block (correctly) renders nothing. To demo on a real product, additively seed approved reviews (≥3) + category attribute defs for that handle — don't reseed (shop-scoped wipe). Summary lazy-generates on first fetch, or force it via `getOrGenerateSummary(shop, id, scope, filters, true)` run under `tsx --env-file=.env`.

28. **Auto-create on unknown handle is a write from a GET loader.** `resolveProductForShop` creates a Product row on first storefront hit; two blocks on the same new product page (e.g. ai-summary + filter-chips) call it concurrently and both miss the `findUnique`, so the losing `create` throws Prisma `P2002` on `@@unique([shop, slug])` → uncaught 500. Guard the create: catch P2002, re-`findUnique`, return that row (or `upsert` with empty `update`). Also means crawlers can create bare rows for typo'd handles — accepted tradeoff, see STATE.

## Phase 6 traps (media, order linkage, review-request email — 2026-07-16)

31. **This Neon DB has pre-existing migration-history drift** (an untracked `Product.shopifyProductId` column with no matching migration file, predating Phase 6) that makes `prisma migrate dev`/`db push --accept-data-loss` refuse to run non-destructively for ANY new schema change, not just the one that first exposed it. Every Phase 6 slice (A, B, C) hit this. Don't waste time retrying migrate-dev flags — go straight to a hand-written migration SQL file (nullable-add → backfill if there's existing data → enforce NOT NULL), apply manually via `psql "$DATABASE_URL" -f ...`, then `npx prisma migrate resolve --applied <name>`. Resolving the underlying drift is still an open TODO (see STATE Next).
32. **`npx <cmd>` fails under the `rtk hook claude` PreToolUse rewrite** (`[rtk: No such file or directory]`) even though `rtk` itself is installed and working (`rtk gain`, `rtk --version` succeed) — the hook's rewrite of the specific `npx` invocation is what breaks. Faster fix than #27's "call the binary directly": prefix the whole command with `rtk proxy` (e.g. `rtk proxy npx prisma migrate resolve --applied <name>`) — it executes raw, unfiltered, and works every time this came up this session.
33. **Presigned-upload flows need three specific safeguards or they're silently exploitable**, all missed on the first pass and only caught by a dedicated diff-review, never by typecheck/tests: (a) sign `ContentLength` into the presigned PUT so a client can't declare a small size then stream a much bigger file; (b) validate a submitted media URL by parsed **origin**, not `url.startsWith(base)` — a suffixed domain (`r2.dev.evil.com`) passes a naive prefix check; (c) track every presigned upload in a "pending/claimed" table (deleted once a review claims it) so a client can't attach someone else's — or an already-used — public storage URL to a new review with no actual upload having happened.
34. **GDPR redaction transaction ordering matters when a new child table has an `ON DELETE RESTRICT` FK** into a model `redactShop` already deletes. The child's `deleteMany` must come BEFORE the parent's in the same `$transaction` array; reversed order throws the FK constraint and rolls back the *entire* shop-erasure transaction — silently breaking compliance for any shop with rows in the new table. No test caught this (none exercised a shop with both parent and child rows present) — only a diff-review reading the transaction array in FK order found it. Check this every time a new table FKs into an already-redacted model.
35. **Shopify order-webhook line items: `line_item.admin_graphql_api_id` is the line item's OWN GID, never the product's.** Deriving a product GID from it (instead of `line_item.product_id`) silently stores the wrong ID for virtually every real order webhook while still passing mocked tests (mocks happened to use the product's id in that field). Always build product GID as `gid://shopify/Product/${line_item.product_id}`; never read `admin_graphql_api_id` off a line item expecting a product reference.

36. **WhatsApp Cloud API webhook signature verification needs raw request body bytes.** X-Hub-Signature-256 HMAC must be computed over the RAW request body before any JSON parsing, because re-serializing the parsed JSON changes bytes (key order, whitespace) and breaks verification. Meta's GET verification handshake carries no shop/tenant context; with per-shop derived verify tokens, scan all connections and match any (documented cheap one-time-setup tradeoff). Shopify's Polaris web components don't support some intuitive props: s-text-field has no `input type="password"` mask, s-select uses `value` not `defaultValue`, s-badge `tone` has no "subdued"/"neutral" per the typed API. Verify component props against installed types, do not assume React-Polaris parity.
